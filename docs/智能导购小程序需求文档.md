## 2. 项目背景

小商家通常已经有自己的商品数据和商品详情页，但用户进入商家小程序后，往往需要自行浏览商品分类、搜索商品，或者通过人工咨询才能找到合适商品。

本项目希望为小商家提供一个轻量级智能导购入口：

1. 商家商品数据由技术侧接入并入库；
2. 系统按商家维度形成商品知识库；
3. 平台为每个商家生成专属太阳码；
4. 用户扫描商家太阳码后，进入该商家的智能导购对话；
5. 用户通过自然语言咨询商品；
6. 系统根据当前商家的商品知识库推荐商品卡片；
7. 用户点击商品卡片后，跳转到商家自有小程序的商品详情页；
8. 后续下单、支付、履约、售后等流程由商家自有小程序承接。

本系统定位为：

```text
智能导购入口，不是交易平台。
```

## 3. 项目目标

### 3.1 核心目标

构建一个面向小商家的智能导购小程序，让用户可以通过扫码和自然语言对话，快速找到某个商家的商品。

### 3.2 MVP 目标

一期重点验证以下闭环：

```text
商家商品数据入库
    ↓
形成商家商品知识库
    ↓
生成商家太阳码
    ↓
用户扫码进入商家导购对话
    ↓
用户自然语言咨询商品
    ↓
AI 推荐商品卡片
    ↓
用户点击商品卡片
    ↓
跳转商家自有小程序商品详情页
```

### 3.3 产品边界

本系统只负责：

- 商家导购入口；
- 商品知识库检索；
- 智能问答；
- 商品推荐；
- 商品卡片展示；
- 跳转商家自有小程序商品详情页；
- 保存用户与商家的历史会话和消息。

本系统不负责：

- 商品详情承载；
- 加购物车；
- 下单；
- 支付；
- 配送；
- 售后；
- 退款；
- 人工客服接待；
- 扫码、点击、推荐等额外行为日志持久化；
- 统计看板和统计接口。

### 3.4 一期已确认边界

| 事项 | 一期结论 |
|---|---|
| 未登录首页 | 允许进入首页，但不展示历史会话 |
| 进入会话 | 必须完成微信授权登录 |
| 游客会话 | 不创建游客会话，不保存游客消息 |
| 商家小程序跳转 | 当前已确认可以跳转第三方小程序，无需再作为待验证风险 |
| 管理后台页面 | 一期不做后台管理页面 |
| 内部管理能力 | 需要提供创建商户、商品导入、会话查看能力 |
| 统计能力 | 一期不做统计接口和统计看板 |
| 持久化数据范围 | 基础业务表保留用户、商家、商品；会话数据仅保存会话和消息；不保存额外行为日志 |
| 商品链接或知识库失效策略 | 一期先不考虑主动失效策略，其他不确定项标记为待确认 |

## 4. 用户角色

### 4.1 C 端用户

C 端用户通过扫码或直接打开小程序使用智能导购。

用户可以：

- 微信授权登录；
- 扫描商家太阳码；
- 登录后进入指定商家的导购对话；
- 登录后直接打开小程序查看历史会话；
- 未登录时直接打开小程序首页，但看不到历史会话；
- 咨询商品；
- 查看 AI 推荐的商品卡片；
- 点击商品卡片跳转商家自有小程序商品页。

### 4.2 商家

商家是商品数据和商品详情页的提供方。

商家需要提供：

- 商家基础信息；
- 商品信息；
- 商品图片；
- 商品价格；
- 商品分类；
- 商品标签；
- 商品描述；
- 商品小程序跳转链接。

### 4.3 平台技术 / 运营人员

平台技术或运营人员负责：

- 商家信息维护；
- 商品数据接入；
- 商品数据清洗；
- 商品知识库构建；
- 商家太阳码生成；
- 会话记录查看；
- 异常数据处理。

## 5. 页面范围

一期小程序主要包含 3 个核心页面：

| 页面 | 说明 |
|---|---|
| 授权登录页 | 用户进入商家会话前未授权时展示，仅做微信授权登录，不做手机号登录 |
| 首页 / 历史会话页 | 用户直接打开小程序时展示；已登录展示历史商家会话，未登录不展示历史会话 |
| 商家导购对话页 | 用户登录后扫码或点击历史会话进入，针对某个商家进行商品导购 |

不设计底部导航。

原因：

- 当前核心页面较少；
- 用户主要路径是扫码进入商家导购；
- 首页只承担历史会话与扫码入口；
- 底部导航会增加不必要的复杂度；
- 小程序整体应保持轻量、直接。

## 6. 核心业务流程

### 6.1 用户首次进入流程

```text
用户打开小程序
    ↓
判断是否已授权
    ↓
未授权：进入首页未登录态
    ↓
不展示历史会话
    ↓
用户点击登录或尝试进入商家会话
    ↓
进入授权登录页
    ↓
微信授权成功
    ↓
获取 openId / unionId / 昵称 / 头像
    ↓
进入首页或指定商家导购页
```

说明：

- 不需要手机号登录；
- 不需要短信验证码；
- 不需要密码登录；
- 用户身份以微信 openId 为主；
- 如需要跨应用统一用户，可使用 unionId；
- 未登录用户可以进入首页，但不能查看历史会话，不能进入商家导购对话，不能发送消息。

### 6.2 用户扫码进入流程

```text
用户扫描商家太阳码
    ↓
打开智能导购小程序
    ↓
小程序解析 scene 参数
    ↓
获取 merchantId
    ↓
判断用户是否已授权
    ↓
未授权：进入授权登录页，并保留扫码来源 merchantId
    ↓
授权成功后继续处理 merchantId
    ↓
已授权：查询 userId + merchantId 是否已有会话
    ↓
已有会话：进入该商家历史会话
    ↓
无会话：创建新会话
    ↓
进入商家导购对话页
```

### 6.3 用户直接打开小程序流程

```text
用户直接打开小程序
    ↓
判断是否已授权
    ↓
未授权：进入首页未登录态
    ↓
不展示历史会话，引导扫码或登录
    ↓
已授权：进入首页
    ↓
展示历史会话列表
```

首页展示的是已登录用户咨询过的商家会话。

一个商家对应一条会话。

### 6.4 用户咨询商品流程

```text
用户在商家导购页输入问题
    ↓
系统识别当前会话所属商家
    ↓
只检索该商家的商品知识库
    ↓
结合当前问题和最近若干条消息
    ↓
生成导购回复
    ↓
返回商品推荐卡片
    ↓
用户点击商品卡片
    ↓
跳转商家自有小程序商品详情页
```

## 7. 关键设计决策

### 7.1 一个用户 + 一个商家 = 一个固定会话

一期采用：

```text
userId + merchantId = conversation
```

即同一个用户多次进入同一个商家时，默认复用同一个会话。

示例：

```text
用户 A + 小林蛋糕店 = 一个固定会话
用户 A + 花田鲜花店 = 一个固定会话
用户 B + 小林蛋糕店 = 另一个固定会话
```

这样做的好处：

- 首页会话列表更清晰；
- 一个商家只展示一条会话；
- 用户再次扫码同一个商家时可以继续上次咨询；
- 避免每次扫码都创建重复会话；
- 数据模型更简单。

### 7.2 AI 不读取完整长期历史

虽然用户和商家之间是固定会话，但 AI 每次回复时不能读取完整历史消息。

一期采用轻量上下文策略：

```text
当前用户问题
+ 当前商家信息
+ 当前商家商品知识库检索结果
+ 最近 10～20 条对话消息
```

不传入该用户与该商家的全部历史消息。

原因：

- 避免旧需求影响当前导购；
- 避免长期会话上下文过长；
- 降低模型调用成本；
- 提高商品推荐准确性；
- 保证当前问题优先。

例如：

用户上周问：

```text
朋友生日有什么蛋糕推荐？
```

本周又问：

```text
你家有抹茶的蛋糕吗？
```

系统应优先理解为：

```text
查询抹茶相关蛋糕
```

不能继续围绕“生日蛋糕”推荐。

### 7.3 商品跳转到商家自有小程序

商品卡片点击后，不在本系统内承接商品详情，而是跳转到商家自己的小程序商品详情页。

商品入库时必须保存：

```text
商品详情页 path
商品详情页参数
```

商家小程序 AppID 维护在商家表中，一期商品表不重复维护 AppID。

本系统只负责导购推荐和跳转。

跳转后的流程由商家小程序负责，包括：

- 商品详情展示；
- 加购；
- 下单；
- 支付；
- 配送；
- 售后。

### 7.4 不设计底部导航

一期不设计底部导航。

页面结构保持轻量：

```text
授权登录页
首页 / 历史会话页
商家导购对话页
```

首页通过按钮进入扫码。

对话页通过返回按钮回到首页。

### 7.5 不做手机号登录

授权登录页只提供微信授权登录。

不包含：

- 手机号输入框；
- 短信验证码；
- 密码登录；
- 账号密码登录。

授权主要用于：

- 获取用户唯一身份；
- 保存历史会话；
- 按用户维度展示历史商家；
- 用户再次扫码时继续上次会话。

## 8. 页面需求

### 8.1 授权登录页

#### 8.1.1 页面定位

授权登录页用于用户首次进入小程序或未授权时完成微信授权。

页面应简洁、清晰，不做复杂登录体系。

#### 8.1.2 页面内容

页面包含：

- 顶部自定义导航；
- 页面标题：智能导购；
- 简洁插画，例如机器人或购物导购形象；
- 主标题：欢迎使用智能导购；
- 副标题：授权后可保存历史会话，快速进入商家导购对话；
- 功能说明卡片；
- 微信授权登录按钮；
- 暂不登录入口，返回首页未登录态；
- 用户协议与隐私政策说明。

#### 8.1.3 页面文案

主标题：

```text
欢迎使用智能导购
```

副标题：

```text
授权后可保存历史会话，快速进入商家导购对话
```

功能说明：

```text
保存历史会话
自动保存你的咨询记录，随时查看

快速进入商家对话
一键继续上次的咨询内容

安全可靠
严格保护你的隐私信息
```

主按钮：

```text
微信授权登录
```

次按钮：

```text
暂不登录，先看看
```

协议文案：

```text
登录即代表你已同意《用户协议》与《隐私政策》
```

#### 8.1.4 交互规则

| 操作 | 结果 |
|---|---|
| 点击微信授权登录 | 调起微信授权，成功后进入首页或扫码来源商家对话页 |
| 点击暂不登录，先看看 | 进入首页未登录态，不展示历史会话，不能进入商家导购对话 |
| 点击用户协议 | 打开用户协议页面 |
| 点击隐私政策 | 打开隐私政策页面 |

#### 8.1.5 未登录态说明

如果用户选择暂不登录：

- 可以进入首页未登录态；
- 不展示历史会话；
- 可以点击扫一扫商家码；
- 扫码后如需进入商家导购对话，必须先完成微信授权登录；
- 不创建游客会话；
- 不保存游客消息；
- 不做游客会话与登录会话的合并逻辑。

### 8.2 首页 / 历史会话页

#### 8.2.1 页面定位

首页是用户直接打开小程序后的默认页面。

首页核心作用：

```text
查看历史商家会话 + 扫描新的商家码
```

未登录用户也可以进入首页，但首页不展示历史会话。

首页不是商城首页，也不是商品流页面。

#### 8.2.2 页面结构

首页包含：

1. 自定义顶部导航区域；
2. 统一平台背景图 / 插画背景；
3. 页面标题；
4. 搜索框；
5. 历史会话列表；
6. 扫一扫商家码按钮；
7. 无会话缺省状态。

#### 8.2.3 顶部自定义导航

首页顶部需要做自定义导航，不使用纯空白顶部。

顶部是平台统一导购入口，不突出某一个商家。

建议风格：

- 使用浅橙色背景；
- 使用 `#ff6900` 作为主视觉点缀；
- 可加入轻量插画元素，如购物袋、对话气泡、定位点；
- 页面主标题突出“智能导购”；
- 副标题表达产品价值。

推荐文案：

```text
智能导购
随时咨询，轻松选购
```

#### 8.2.4 搜索框

搜索框用于搜索用户咨询过的商家。

占位文案：

```text
搜索咨询过的商家
```

搜索框旁边不需要“帮我推荐”按钮。

#### 8.2.5 历史会话列表

历史会话按用户维度展示。

一个商家最多展示一条会话。

展示规则：

| 用户状态 | 展示结果 |
|---|---|
| 已登录 | 展示当前用户的历史商家会话 |
| 未登录 | 不展示历史会话，展示未登录缺省态和登录引导 |

排序规则：

```text
按最近消息时间倒序排列
```

列表字段：

| 字段 | 说明 |
|---|---|
| merchantLogo | 商家头像 / Logo |
| merchantName | 商家名称 |
| lastMessage | 最近一条消息摘要 |
| lastMessageTime | 最近消息时间 |
| conversationId | 会话 ID |
| merchantId | 商家 ID |

示例数据：

| 商家 | 最近消息 | 时间 |
|---|---|---|
| 小林蛋糕店 | 你家有抹茶蛋糕吗？ | 10:24 |
| 花田鲜花店 | 有没有适合生日送的花？ | 昨天 16:30 |
| 老张水果铺 | 有榴莲吗？ | 昨天 11:08 |
| 阿南咖啡 | 有没有低糖拿铁？ | 前天 15:42 |

#### 8.2.6 会话点击规则

用户点击某条历史会话：

```text
进入对应商家的导购对话页
```

进入后继续使用该用户与该商家的固定会话。

#### 8.2.7 扫一扫商家码按钮

首页底部提供主按钮：

```text
扫一扫商家码
```

点击后调起微信扫码能力。

扫码成功后：

```text
解析商家码 → 获取 merchantId → 进入对应商家导购页
```

如果用户未登录：

```text
解析商家码 → 获取 merchantId → 进入授权登录页 → 授权成功 → 进入对应商家导购页
```

#### 8.2.8 无历史会话缺省状态

当用户没有历史会话时，展示缺省状态。

推荐文案：

```text
暂无导购会话
扫描商家的导购码，开始智能选购
```

可提供按钮：

```text
扫一扫商家码
```

未登录时展示未登录缺省态。

推荐文案：

```text
登录后可查看历史导购会话
也可以先扫描商家的导购码，授权后开始咨询
```

### 8.3 商家导购对话页

#### 8.3.1 页面定位

商家导购对话页用于承载用户与某个商家的智能导购对话。

该页面必须突出商家身份，让用户明确知道当前正在咨询哪个商家。

#### 8.3.2 页面结构

页面包含：

1. 自定义商家顶部导航；
2. 商家信息区；
3. 商家背景图 / 商品氛围图；
4. 推荐提问条；

## 9. 商品知识库需求

### 9.1 知识库定义

每个商家的商品数据形成独立商品知识库。

知识库按商家隔离。

用户进入某个商家的导购页时，只能查询该商家的商品。

不能推荐其他商家的商品。

### 9.2 商品知识库内容

商品知识库包含：

- 商品名称；
- 商品描述；
- 商品分类；
- 商品标签；
- 商品价格；
- 商品图片；
- 商品规格；
- 商品卖点；
- 商品适用场景；
- 商品适用人群；
- 商品上下架状态；
- 商品库存状态；
- 商品小程序跳转信息。

### 9.3 商品检索能力

系统需要支持自然语言商品检索。

示例：

| 用户问题 | 系统理解 |
|---|---|
| 你家有抹茶的蛋糕吗？ | 查询关键词：抹茶、蛋糕 |
| 有没有适合生日的蛋糕？ | 查询场景：生日 |
| 有没有便宜一点的？ | 查询价格偏好：低价 |
| 有适合小朋友吃的吗？ | 查询人群：儿童 |
| 有没有不太甜的？ | 查询口味：低甜、低糖 |
| 下午茶有什么推荐？ | 查询场景：下午茶 |

### 9.4 检索结果要求

检索结果需要返回：

- 商品 ID；
- 商品名称；
- 商品图片；
- 商品价格；
- 商品标签；
- 商品描述；
- 商品跳转链接；
- 匹配原因；
- 相关度分数；
- 商品状态。

### 9.5 商品推荐原则

推荐优先级建议：

1. 当前商家商品；
2. 上架商品；
3. 有库存商品；
4. 与用户当前问题高度匹配；
5. 商品名称匹配；
6. 商品标签匹配；
7. 商品分类匹配；
8. 商品描述匹配；
9. 场景匹配；
10. 热销或推荐商品。

## 10. AI 回复策略

### 10.1 基本原则

AI 必须遵守：

- 只回答当前商家的商品；
- 不能编造不存在的商品；
- 不能推荐其他商家的商品；
- 优先基于商品知识库结果回答；
- 无商品匹配时明确说明没有找到；
- 不承诺库存、配送、支付、售后等非本系统能力；
- 商品价格、名称、跳转链接必须来自商品库。

### 10.2 AI 上下文组装

每次 AI 调用时，组装上下文：

```text
当前商家信息
当前用户问题
最近 10～20 条消息
当前商家商品检索结果
必要的系统规则
```

不要传入完整长期历史。

### 10.3 有结果回复

用户问：

```text
你家有抹茶的蛋糕吗？
```

如果有匹配商品，回复：

```text
有的，下面几款抹茶相关蛋糕你可以看看：
```

并展示商品卡片。

### 10.4 无结果回复

如果没有匹配商品，回复：

```text
暂时没有找到相关商品，你可以换个关键词试试。
```

如果有相似商品，可以回复：

```text
暂时没有找到完全匹配的商品，不过下面这些口味或场景比较接近，你可以看看：
```

注意：

- 必须明确说明不是完全匹配；
- 不能把相似商品说成用户明确要找的商品。

### 10.5 非商品问题回复

如果用户问和商品无关的问题，例如：

```text
你们几点关门？
```

如果商品知识库没有商家营业信息，则回复：

```text
我暂时只能帮你查询和推荐本店商品，营业时间建议进入商家小程序查看。
```

## 11. 商家太阳码需求

### 11.1 太阳码用途

每个商家拥有一个专属太阳码。

用户扫描后进入该商家的智能导购对话页。

### 11.2 太阳码参数

太阳码需要携带商家标识。

建议使用微信小程序 scene 参数。

示例：

```text
scene=m_xxx
```

后台解析后得到：

```text
merchantId=xxx
```

### 11.3 扫码处理逻辑

扫码后：

1. 解析 scene；
2. 获取 merchantId；
3. 校验商家是否存在；
4. 校验商家是否启用；
5. 判断用户是否授权；
6. 未授权则进入授权登录页，并保留当前 merchantId；
7. 授权成功后，查询 userId + merchantId 是否已有会话；
8. 有会话则进入历史会话；
9. 无会话则创建会话；
10. 进入商家导购对话页。

### 11.4 太阳码异常

| 场景 | 提示 |
|---|---|
| 无法解析 scene | 该导购码无效 |
| merchantId 不存在 | 未找到对应商家 |
| 商家已停用 | 该商家导购服务暂不可用 |
| 网络异常 | 网络异常，请稍后再试 |

简约、轻量、温暖、清晰、导购感

- 不放“帮我推荐”按钮。

首页底部：

- 主按钮为“扫一扫商家码”；

## 17. 权限与数据隔离

### 17.1 用户隔离

用户只能看到自己的历史会话。

查询会话时必须带 userId。

未登录用户不能查询历史会话，不能进入商家导购对话，不能发送消息。

### 17.2 商家隔离

用户在某个商家会话中，只能查询该商家的商品。

商品检索必须带 merchantId。

禁止跨商家推荐商品。

### 17.3 商品跳转安全

商品跳转信息必须来自商品库。

商家小程序 AppID 必须来自商家表。

不能由 AI 自行生成跳转链接。

跳转前建议校验：

- 商品是否存在；
- 商品是否上架；
- 商家是否启用；
- 商家表中的 `miniProgramAppId` 是否有效；
- path 是否存在。

## 19. MVP 验收标准

### 19.1 授权登录

- 用户首次进入可看到授权登录页；
- 页面不出现手机号登录；
- 页面不出现短信验证码；
- 用户可以微信授权登录；
- 用户可以选择暂不登录，先看看；
- 暂不登录后进入首页未登录态；
- 未登录用户看不到历史会话；
- 未登录用户不能进入商家导购对话；
- 授权后可以保存历史会话。

### 19.2 首页

- 用户直接打开小程序后进入首页；
- 首页无底部导航；
- 首页顶部有自定义背景，不是纯空白；
- 首页展示搜索框；
- 搜索框不包含“帮我推荐”按钮；
- 已登录首页展示历史会话列表；
- 未登录首页不展示历史会话；
- 一个商家最多展示一条历史会话；
- 用户可以点击历史会话进入商家导购页；
- 用户可以点击“扫一扫商家码”扫码；
- 无历史会话时展示缺省状态。

### 19.3 商家导购页

- 用户扫码商家码后可以进入对应商家的导购页；
- 未登录用户扫码后先进入授权登录页，授权成功后进入对应商家的导购页；
- 对话页顶部突出商家身份；
- 不同商家可以展示不同头像、名称、背景图；
- 页面展示推荐提问；
- 用户可以发送商品咨询；
- AI 可以返回文本回复；
- AI 可以返回商品卡片；
- 商品卡片信息来自商品库；
- 商品卡片点击后跳转商家自有小程序商品详情页。

### 19.4 内部管理能力

- 平台内部可以创建商户；
- 平台内部可以导入商户商品；
- 商品导入后可以形成或更新商品知识库；
- 平台内部可以查看会话和消息；
- 一期不提供后台管理页面；
- 一期不提供统计接口和统计看板。

### 19.5 会话机制

- 一个用户 + 一个商家只能有一个固定会话；
- 用户多次扫码同一商家时复用历史会话；
- 用户直接进入首页时按商家展示历史会话；
- 不创建游客会话；
- 不保存游客消息；
- AI 不读取完整历史消息；
- AI 每次只读取最近 10～20 条消息和当前商品检索结果。

### 19.6 商品推荐

- AI 只能推荐当前商家的商品；
- AI 不能编造商品；
- 下架商品不能推荐；
- 商品价格、图片、跳转链接必须来自商品库；
- 无匹配商品时有明确兜底回复。

## 20. 技术注意事项

### 20.1 小程序跳转能力

当前已确认平台小程序可以跳转第三方商家小程序。

一期按以下方案实现：

- 商品卡片点击后直接跳转商家自有小程序商品详情页；
- 跳转 AppID 必须来自商家表；
- 跳转 path 和参数必须来自商品库；
- 不在本小程序承载商品详情；
- 不再把第三方小程序跳转作为待验证风险。

### 20.2 商品数据同步

一期由商户提供商品数据，平台内部通过导入能力入库。

一期需要支持：

- 创建商户；
- 导入商户商品；
- 商家表维护商家小程序 AppID；
- 商品导入时写入商品详情页 path 和参数；
- 导入后形成或更新商品知识库。

后续可支持：

- API 同步；
- 商家后台维护；
- 定时同步；
- 增量更新。

### 20.3 商品知识库更新

商品发生变化后，需要更新知识库。

触发场景：

- 商品新增；
- 商品下架；
- 商品价格变更；
- 商品描述变更；
- 商品标签变更；
- 商品链接变更。

一期先不考虑主动失效策略。

说明：

- 不做知识库 TTL；
- 不做商品链接有效期管理；
- 不做自动失效扫描；
- 其他未明确的失效场景标记为待确认。

### 20.4 AI 检索方式

建议采用：

```text
关键词检索 + 向量检索 + 商品状态过滤
```

检索时必须带：

```text
merchantId
```

避免跨商家召回商品。
